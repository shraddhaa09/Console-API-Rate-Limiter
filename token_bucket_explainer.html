Problem It Solves

Why is rate limiting needed, and what specific gap does Token Bucket fill?

In any web API, a single bad actor (or buggy code) can flood the server with thousands of requests per second. Without control, this causes the entire server to slow down or crash for ALL users — a Denial of Service (DoS) scenario.

The simplest fix is the Fixed Window algorithm — count requests per minute, reset every minute. But this has a dangerous loophole called the Boundary Burst Bug:

💥 The Boundary Burst Problem

Rule: Allow max 10 requests per 60 seconds.

— Attacker sends 10 requests at second 59 (end of Window 1) → all ALLOWED ✓
— Window resets at second 60
— Attacker sends 10 requests at second 61 (start of Window 2) → all ALLOWED ✓

Result: 20 requests passed in just 2 seconds! The server sees this as valid but it is effectively a mini-attack.

Token Bucket solves this by removing the concept of a sharp "reset boundary" entirely. Instead of counting in fixed time blocks, it refills tokens smoothly over time. There is no magic moment to exploit.

Concept Overview

The core idea in plain English.

The Token Bucket algorithm imagines every user having an invisible bucket. This bucket:

  • 🪣 Has a maximum capacity (e.g. 10 tokens)
  • 🔄 Continuously refills at a fixed rate (e.g. 2 tokens per second)
  • 🎫 Every request consumes 1 token from the bucket
  • ✅ If tokens > 0 → request is allowed
  • ❌ If bucket is empty → request is denied
  • 🚫 If bucket is full → new tokens overflow (are discarded)
🎯 Key Insight: A user who has been idle for a while saves up tokens and can then send a burst of requests. But once they use those saved tokens, they must wait for the bucket to refill. This perfectly mirrors real-world "burst then slow" usage patterns.

Real-World Analogy

A picture worth a thousand lines of code.

The Physical Bucket Analogy

Picture a real physical bucket sitting on a shelf. Above it is a tap that drips water at a constant rate — say, 2 drops per second. The bucket can hold a maximum of 10 drops.

Every time you want to send an API request, you must take one drop out of the bucket. The rules are:

  • If there is water (tokens) in the bucket → you can take a drop → request goes through
  • If the bucket is empty → you must wait for the tap to drip more → request is blocked
  • If you leave the bucket alone and come back later → it has refilled — you can burst again
  • If it rains and the bucket overflows → extra water is wasted (tokens are capped at capacity)
🧠 Second Analogy — Highway Token Booth:
Imagine a highway toll with tokens. Every car (request) needs 1 token to pass. Tokens are replenished at a fixed rate. A car that arrives when no tokens are available is turned away. There is no "midnight reset" — the booth just quietly refills all day.

Why We Selected This Algorithm

Justification for including Token Bucket in our C++ Rate Limiter project.

We selected Token Bucket as Algorithm 1 for several strong reasons that make it the ideal starting algorithm for our project:

  1. Industry standard — It is what AWS and Stripe actually use. Understanding it first gives context for why other algorithms exist.
  2. Simple to implement — Requires only 4 variables per user and a few lines of arithmetic. No complex data structures needed.
  3. Bursting support — Allows a user to burst (use saved tokens) then naturally slows them down — this is realistic user behavior.
  4. Fixes boundary burst — Unlike Fixed Window, there is no reset moment to exploit. The smooth refill eliminates the bug.
  5. O(1) in everything — Both time and space are constant, meaning it can handle thousands of concurrent users efficiently.
  6. Good baseline for comparison — Once understood, every other algorithm in the project makes more sense by comparison.

How the Algorithm Works

The complete internal mechanics, from start to finish.

Full Step-by-Step Walkthrough
  1. Initialization — When a user first appears, create their bucket with tokens = capacity (full bucket), last_refill = now().
  2. Request arrives — A client sends a request. The system retrieves (or creates) that client's bucket state.
  3. Compute elapsed time — elapsed = current_time - last_refill. This is how many seconds have passed since we last added tokens.
  4. Refill tokens — tokens = tokens + (elapsed × refill_rate). Add all the tokens that should have appeared during the gap.
  5. Cap the tokens — tokens = min(tokens, capacity). You cannot exceed the bucket size. Extra tokens are discarded.
  6. Update refill time — last_refill = current_time. Next time we refill, we measure from NOW.
  7. Check availability — Is tokens >= 1.0? If yes: subtract 1 and return ALLOW. If no: return DENY.
🔑 Key Design Choice: We use lazy refill — tokens are NOT added on a separate timer thread. Instead, they are calculated on-demand when a request arrives. This avoids needing background threads and keeps the design stateless between requests.

Start State of the Algorithm

What is initialized when the rate limiter is first set up for a user.

Variables Initialized Per User
VariableTypeInitial ValueWhat It Stores
tokens double = capacity (full) Current number of tokens in the bucket right now
capacity double e.g. 10.0 Maximum number of tokens the bucket can hold
refill_rate double e.g. 2.0 How many tokens are added per second
last_refill time_point = now() The exact timestamp when we last calculated a refill
📝 Why double (not int)? Refills happen continuously over time. If refill_rate = 0.5 tokens/sec and 1 second passes, you earn 0.5 tokens — not 0. Using double allows fractional tokens to accumulate until a whole token is available.

What Happens During Each Request

The exact sequence of operations every time allow_request() is called.

// ── Called every time user sends a request ── FUNCTION allow_request(user_id): bucket = get_bucket(user_id) // fetch this user's state now = get_current_time() // Step 1: How much time has passed since last refill? elapsed = now - bucket.last_refill // in seconds (e.g. 3.5) // Step 2: Refill tokens proportionally to elapsed time bucket.tokens = bucket.tokens + (elapsed × bucket.refill_rate) // Step 3: Cap — cannot exceed max capacity bucket.tokens = min(bucket.tokens, bucket.capacity) // Step 4: Stamp the current time as last refill bucket.last_refill = now // Step 5: Is there at least 1 token available? IF bucket.tokens >= 1.0: bucket.tokens = bucket.tokens - 1.0 // consume a token RETURN ALLOW ✅ ELSE: RETURN DENY ❌

Allow & Deny Conditions

Exactly when a request passes and when it is blocked.

✅ Request Is ALLOWED When:
  • Bucket has tokens ≥ 1.0 after refill calculation
  • User has been idle and tokens have accumulated
  • Request arrives right after initialization (bucket is full)
  • User's request rate is slower than the refill rate
  • A burst is within the saved-up token capacity
❌ Request Is DENIED When:
  • Bucket is empty: tokens < 1.0
  • User sent requests faster than the refill rate for too long
  • Burst exceeded the capacity limit
  • Attacker is hammering the API continuously
  • User is doing a "drip attack" faster than refill speed
⏳ What should a denied user do? They must wait for tokens to refill. Minimum wait time = 1.0 / refill_rate seconds. With refill_rate = 2.0, they must wait at least 0.5 seconds before their next request can succeed.

End State / Continuous Operation

The algorithm never "ends" — it keeps running for the lifetime of the server.

Unlike Fixed Window which has a hard reset every N seconds, the Token Bucket is stateful and continuous. Each user's bucket state persists between requests. Here is what "steady state" looks like:

  • 🕐 Tokens accumulate automatically as real clock time passes
  • 📥 Every request triggers a refill calculation then a token check
  • 🧹 Idle user buckets can be garbage collected after inactivity timeout
  • ♾️ The algorithm handles millions of users simultaneously — each user has an independent bucket with 4 variables

Data Structures Used

What internal structures store the algorithm's state and why.

StructureUsed ForWhy This One?
struct TokenBucket Holds the 4 state variables per user Bundles related data together cleanly; zero overhead
std::unordered_map<string, TokenBucket> Maps each client_id → their bucket state O(1) average lookup by user ID; hash map is fastest option
double (tokens, capacity, refill_rate) Stores fractional token counts Allows sub-second precision; int would cause rounding errors
chrono::time_point Stores last_refill timestamp High-resolution clock; allows sub-millisecond calculations
📌 No queues, no trees, no logs. That is the beauty of Token Bucket. It uses only primitive types and a hash map. This is the main reason it scales to millions of users — the memory footprint per user is essentially just 4 numbers (≈ 32 bytes).

Time & Space Complexity

How the algorithm scales with users and traffic.

⏱ Time Complexity
O(1)
Every request takes the exact same number of operations regardless of how many users exist or how many requests came before it. No loops, no sorting, no searching.
💾 Space Complexity
O(U)
U = number of unique users. Each user stores exactly 4 variables (≈ 32 bytes). Memory grows linearly with users but not with traffic volume.
Why O(1) Matters in Production

Imagine 100,000 requests per second hitting the server. With O(1) per request, the rate limiter adds the same tiny overhead whether it is request #1 or request #100,000,000. There is no performance cliff. This is why AWS chose this algorithm for their API Gateway at global scale.

Advantages & Disadvantages

Honest trade-off analysis — no algorithm is perfect.

✓ Advantages
  • Extremely simple to implement — just arithmetic
  • O(1) time and space — scales to any traffic volume
  • Allows natural bursting (saved tokens can be spent)
  • No reset boundary — immune to boundary burst attack
  • Smooth traffic shaping — no sudden allow/deny cliffs
  • Used by AWS and Stripe — battle-tested at global scale
  • Works well with fractional rates (e.g. 0.5 req/sec)
✗ Disadvantages
  • Drip attack vulnerability — attacker sends exactly 1 req per refill interval and is never denied
  • Harder to reason about than Fixed Window for non-technical users
  • Float precision may cause minor inconsistencies at extreme scale
  • Does not track request history — cannot detect slow distributed attacks
  • Requires careful tuning of capacity vs refill_rate ratio
🛡️ The Drip Attack: An attacker sends exactly 1 request every (1 / refill_rate) seconds. They consume tokens as fast as they refill. They are never denied, but they send continuous requests forever. This is why the Hybrid Limiter (Algorithm 6) combines Token Bucket with Sliding Window Counter — to catch this pattern.

Clean pseudocode you can translate into any language.

Edge Cases to Consider

Tricky situations that your implementation must handle correctly.

🕐 Long Idle Period
User returns after 10 minutes. Refill calculation gives 1200 tokens. Must cap at capacity (e.g. 10). Without the cap, users could burst indefinitely after being idle.
⚡ Simultaneous Requests
Two threads try to allow requests at the exact same millisecond. Without a mutex, both might read tokens=1 and both allow the request, consuming 2 tokens but only checking once.
🆕 First Request Ever
User not yet in the map. Must create their bucket with tokens = capacity before calling allow_request. Check if (buckets.find(id) == buckets.end()) first.
🔢 Very High Refill Rate
If refill_rate = 1000 and capacity = 10, the bucket fills almost instantly. The cap ensures this doesn't give unlimited tokens. Test extreme values during development.
🧮 Float Precision
At high scale, floating point errors can accumulate. tokens might be 0.999999 instead of 1.0. Use a small epsilon (e.g. tokens >= 0.9999) or round appropriately.
♻️ Memory Cleanup
If 1 million users connect once and leave, the unordered_map grows forever. In a real system, evict buckets that haven't been used in N minutes to prevent memory bloat.

How Token Bucket stacks up against the other 5 algorithms in our project.

Algorithm Memory Speed Accuracy Fixes Burst? Drip Attack? Best For
🪣 Token Bucket O(1) O(1) Good ✓ YES ✗ Vulnerable AWS, Stripe, general APIs
🪟 Fixed Window O(1) O(1) Low (buggy) ✗ NO — Baseline / simple demos
📋 Sliding Window Log O(R) O(1) avg Perfect 100% ✓ YES ✓ Catches it Low traffic, max accuracy
🔢 Sliding Counter O(1) O(1) ~95% ~ Approx ~ Partial Cloudflare, massive scale
🌳 Segment Tree O(T) O(log T) Good ✓ YES ✓ YES DSA showcasing, high traffic
🔀 Hybrid Limiter O(1) O(1) Strictest ✓ YES ✓ YES Payments, login, admin APIs
🏆 When to pick Token Bucket: When you need the simplest, fastest solution that still handles burst traffic fairly. It is the best default choice for most APIs. Its only weakness (drip attack) is addressed by combining it with Sliding Counter in the Hybrid Limiter.
📌 Token Bucket vs Leaky Bucket (common confusion): The Leaky Bucket algorithm is different — it drains requests at a fixed rate and acts like a queue, enforcing strict output pacing. Token Bucket is more flexible — it allows bursting. They solve different problems.
